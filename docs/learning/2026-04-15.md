# Learnings — 2026-04-15

## 2026-04-15 17:23:24 — Large file Read: use Grep or offset/limit instead of reading whole file

**Error**: `File content (37219 tokens) exceeds maximum allowed tokens (25000)` when reading a large tool-result file.

**Solution**: For files over 25K tokens, never attempt a full `Read`. Use `Grep` to search for specific patterns, or use `offset`/`limit` parameters. When the file is a single-line blob (like MCP JSON responses), `Grep` with context lines won't help much — instead use `Bash` with `python3` to parse and extract the relevant section by keyword.

## 2026-04-15 17:23:24 — MCP tool-result files use JS object format, not valid JSON

**Error**: `json.JSONDecodeError` when trying to `json.loads()` on content extracted via regex from an MCP tool-result file. The file uses JS object notation (`{ content: [...] }`) with single quotes, not standard JSON.

**Solution**: Skip JSON parsing entirely. Treat the file as raw text — do simple string operations like `.find()` and slicing after unescaping `\\n` → `\n` and `\\"` → `"`. This is more robust than trying to convert JS object syntax to valid JSON.

## 2026-04-15 17:23:24 — rtk-proxied find/ls may return non-zero exit codes

**Error**: `Exit code 1` from `rtk find ... -o -name ...` even though partial results were returned. The `find` command through rtk proxy can exit non-zero when some paths don't match or have permission issues.

**Solution**: Use the dedicated `Glob` tool for file searches instead of `find` via Bash. When Bash `find` is unavoidable, don't chain with `&&` — use `;` or check output regardless of exit code.

## 2026-04-15 17:29:25 — Expected CLI validation errors are not real failures

**Note**: When testing CLI scripts with intentionally invalid inputs (missing required params, invalid enum values), exit code 1 is expected behavior. The self-improving-light hook captures these as errors, but they're deliberate negative tests confirming error handling works correctly. No action needed — just be aware that validation testing will always trigger the error hook.

## 2026-04-15 17:54:51 — Env vars added to ~/.zshrc are not available in Bash tool

**Error**: `请设置环境变量 AONE_APP_NAME 和 AONE_APP_SECRET` when running a script via Bash tool, even though the vars were just added to `~/.zshrc`.

**Solution**: The Bash tool spawns non-interactive shells that may not source `~/.zshrc`. When env vars were added in the same session, pass them inline: `AONE_APP_NAME=x AONE_APP_SECRET=y node script.mjs`. The user running `! source ~/.zshrc` only affects their local shell, not the Bash tool's subprocess.
