# Tech Research

先拆解问题，再生成关键词，再按强制执行流程完成搜索、采集和写作。

这个 skill 是通用研究执行器，不只服务某一个行业或某一种专题。默认先做研究前处理，再强制进入全链路研究执行。

## 核心目标

把用户输入的模糊主题或长文档，转成一套可执行的研究结构，并按强制执行流程落地，至少包括：

- 主题定义
- 研究边界
- 子问题清单
- 关键词池
- 中英文映射
- 搜索语句
- 后续执行建议

无论用户输入是简单主题、问题描述、会议纪要还是任务文档，只要使用本 skill，就必须基于这套结构继续执行完整研究流程，不能停留在拆题或关键词阶段。

## 核心原则

1. 先理解输入，再生成关键词；不要一上来就机械搜索。
2. 优先补足用户没说清楚但研究必须有的维度，例如对象、场景、时间、地域、行业、角色、约束。
3. 关键词必须分层，不要只给一串平铺词。
4. 同时生成中文关键词和英文关键词，除非用户明确只要单语。
5. 区分“原始输入中的事实”与“根据输入推导出的研究方向”。
6. 所有任务默认都要进入搜索、采集、阅读和报告阶段，不能只输出拆题结果。
7. 所有落盘内容默认保存到 `${OPENCLAW_DATA_DIR}`。

## 输入解析

接到主题、任务说明或文档后，优先提取这些元素：

- 研究主题是什么
- 研究对象是谁
- 研究对象所在行业或场景是什么
- 用户真正想回答的问题是什么
- 成功交付物是什么
- 有哪些显式要求
- 有哪些隐含约束
- 是否包含时间范围、地域范围、语言范围

如果输入是长文档，先压缩成“任务摘要 + 约束摘要 + 线索摘要”三部分。

## 拆题流程

### 1. 定义主问题

把原始输入改写成 1 句主问题，确保问题可研究、可验证、可扩展。

例如不要停留在“研究 AI 提效”，而要写成“研究企业内部各职能如何通过 AI 工具提升流程效率，以及主要厂商和落地方式”。

### 2. 拆成子题

按最适合当前主题的维度拆解，常见维度包括：

- 行业
- 角色
- 场景
- 流程
- 技术
- 产品
- 竞品
- 指标
- 风险
- 地域
- 时间线

拆题结果应让后续搜索或写作可以逐块执行，而不是互相重叠。

### 3. 扩写关键词

围绕每个子题扩写关键词，至少覆盖这些层次：

- 核心主题词
- 同义词和近义表达
- 上下游相关词
- 角色词
- 任务词
- 产品/工具词
- 方法论词
- 结果导向词
- 风险/问题词

如果主题明显带有固定表达模式，例如 `AI+XX`、`XX software`、`XX automation`、`XX workflow`，应主动扩写出对应模式。

### 4. 生成中英文映射

把关键词整理成中英文对照表，避免后续检索时来回翻译。

英文词不要求逐字直译，要优先使用真实世界常见表达。

### 5. 生成搜索语句

基于关键词池生成一组可直接检索的搜索语句，建议分为：

- 宽搜索
- 精确搜索
- 趋势搜索
- 案例搜索
- 产品搜索
- 对比搜索
- 时间限定搜索

如果用户要最近信息，默认在搜索语句里显式带上时间范围。

## 推荐输出结构

默认输出以下内容，不按“轻量任务”裁剪：

1. 主题重述
2. 研究目标与边界
3. 子题拆解
4. 关键词池
5. 中英文映射
6. 可直接使用的搜索语句
7. 后续执行建议

## 研究执行

所有任务在完成拆解后，必须继续进入下一阶段：

1. 先确定输出目录和命名规范。
2. 对每个子题或关键词做资料采集。
3. 保存原始链接、摘要、正文抽取结果和结构化笔记。
4. 素材不足时先补料，再写分析。
5. 报告结尾保留来源链接。

## 强制执行流程

这是本 skill 的默认主流程，不需要等待用户额外说明。只要本 skill 被触发，所有任务都必须按下面流程执行；如果用户输入中还包含明确的任务规格、步骤要求、技能约束或交付规范，则在不削弱下面流程的前提下继续叠加执行，不能只做泛化理解、轻量总结或跳过后续环节。

### 必须使用的技能

- `tavily-search`
- `aisa-search`
- `aisa-twitter`
- `aisa-youtube`
- `video-download`
- `scrapling-official`
- `humanizer-zh`

除非用户明确删除某一项，或某个技能当前不可用且已向用户报告，否则不要擅自跳过。

### 必须执行的步骤

1. 先用 `tavily-search` 技能盘点目标主题下的角色、工种、职责和工作内容。
2. 从角色与工作内容中提炼关键词，并按任务要求整理为目标格式；如果规格要求 `AI+XX`，就必须按 `AI+XX` 生成。
3. 把关键词翻译成英文，并在后续搜索阶段统一使用英文关键词。
4. 遍历关键词，用 `tavily-search` 技能对近 365 天内容做高级深度搜索，每个关键词不少于 20 条记录，并保存结果。
5. 遍历关键词，再用 `tavily-search` 技能对 `reddit.com` 做近 365 天高级深度搜索，每个关键词不少于 20 条记录，并保存结果。
6. 遍历关键词，用 `aisa-search` 技能的 Multi-Stage Orchestration 搜索数据，每个关键词不少于 20 条记录，并保存结果。
7. 遍历关键词，用 `aisa-twitter` 技能搜索趋势和代表性内容，并保存结果。
8. 尽量读取并保存原始链接正文；若可用，优先使用 `scrapling-official` 技能的 extract stealthy-fetch 能力，而不只保留摘要。
9. 遍历关键词，用 `aisa-youtube` 技能搜索相关视频，每个关键词不少于 10 个视频，并输出具体视频链接。
10. 如果没有拿到视频链接，必须先排查原因并解决，不能直接跳过。
11. 对视频链接使用 `video-download` 技能获取字幕，必须带上任务要求的固定参数。
12. 依次阅读并理解每个关键词下的全部素材，包括 `md` 文档、网页正文抽取结果和常见字幕文件；同一视频多种字幕后缀时读取一种即可。
13. 为每个关键词生成对应研究报告；如果规格要求报告长度下限，必须满足。
14. 合并所有关键词报告，生成最终汇总报告；如果规格要求“不是简短总结而是尽量保留原文内容”，则必须按此执行。
15. 使用humanizer-zh技能对内容做优化，去除掉 AI(特别是明显的 chatgpt 风格) 痕迹。

### 必须遵守的约束

- 所有下载和生成文件默认保存到 `${OPENCLAW_DATA_DIR}`。
- 每个课题必须使用独立目录。
- 每个视频必须提取标题，并以精简标题作为目录名保存相关文件。
- 出现报错时必须立即反馈用户，由用户决定是否继续。
- `video-download` 只能由主 Agent 执行；其他技能可以按情况交给子 Agent。
- 最终报告只能在完整阅读该课题素材后生成。
- 报告末尾必须列出使用过的素材网页地址。
- 长任务必须定时汇报进度，开启定时任务，理想频率约为 1 分钟一次。

### 逐子题深度检索

对每个子题执行搜索，收集以下六类信息（缺一不可）：

| 维度 | 要求 |
|------|------|
| 具体公司 | 公司全称、官网、总部、成立时间、核心定位 |
| 真实案例 | 标杆客户名称、应用场景、落地过程 |
| 技术实现细节 | 架构、算法、关键模块、数据流、集成方式 |
| 优势与效果 | 对比竞品的差异化能力 |
| 量化指标 | AHT 下降比例、效率提升百分比、ROI、用户规模等可验证数字 |
| 非常规核心技术 | 不常见但关键的技术手段（如专有模型微调、知识蒸馏、MCP 编排等） |

搜索时优先使用英文关键词获取一手资料，再用中文补充国内案例。

### `video-download` 固定参数

当规格文件要求拉取 YouTube 等视频字幕时，默认按下面参数执行：

```json
{
  "urls": ["https://xxx"],
  "cookiefile": "~/workspace/openclaw-workspace/cookie/www.youtube.com_cookies.txt",
  "onlysubtitle": true,
  "overwrite_subtitle": false,
  "download_subtitle": true
}
```
**如果提示 cookie 无效等错误信息，则去掉cookiefile重新尝试**
**如果用户给了不同参数，以用户明确要求为准；否则不要擅自改动这些默认值，也不要省略字幕步骤。**

## 目录建议

当任务需要落盘时，可使用如下通用结构：

```text
${OPENCLAW_DATA_DIR}/<课题目录>/
  00-任务说明/
  01-输入解析/
  02-子题拆解/
  03-关键词池/
  04-search-results/
  05-source-content/
  06-notes/
  07-reports/
```

按任务复杂度增减目录，不要为了整齐强行创建无用层级。

## 质量门槛

- 关键词要“够多、够准、够可搜”，不是只换几个近义词。
- 子题要能支撑后续搜索和写作，不能只是概念罗列。
- 若输入信息不足，要明确写出假设。
- 只要使用本 skill，就不能只交付拆题、关键词或简短总结，必须完成强制执行流程中的检索、采集、阅读和报告环节；若因工具或外部条件受阻，必须立刻向用户报告阻塞点。
- 若用户给了硬性规格文件，按规格执行；不要用通用模板覆盖用户要求。
